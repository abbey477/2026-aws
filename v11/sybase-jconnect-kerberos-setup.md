# Sybase ASE + jConnect 4 + Kerberos on RHEL — Setup & Validation Guide

A step-by-step playbook for getting jConnect 4 (`jconn4.jar`) to authenticate to Sybase ASE using Kerberos 5 on Red Hat Enterprise Linux. Each step includes a validation check so you can catch problems as early as possible.

**Assumption:** `isql` already authenticates with Kerberos against your Sybase server. If it does, the KDC, DNS, server-side SPN, and your TGT are already working — the remaining work is on the Java/JDBC side.

---

## Prerequisites & Ownership

Before starting, know who owns what:

| Component | Owner | Notes |
|---|---|---|
| `/etc/krb5.conf` | Sysadmin / Kerberos admin team | System-wide config; do not edit on shared hosts without coordination |
| KDC, realms, SPNs | Identity / AD team | Source of truth for realm names and SPN strings |
| Sybase SPN + keytab | Sybase DBA | Provides the exact SPN to use in JDBC URL |
| JAAS config, JDBC URL, app launch flags | Application developer (you) | Lives with your app |

Since `isql` already works, `/etc/krb5.conf` is already correct — **read it, don't rewrite it**.

---

## Step 1 — Confirm the Kerberos ticket cache is healthy

Run as the same Linux user that will run the Java app:

```bash
klist
```

If no cache exists:

```bash
kinit your_user@YOUR.REALM
klist
```

**✅ Validate:** Output shows a TGT for `krbtgt/YOUR.REALM@YOUR.REALM` with a future expiration.

**⚠️ Gotcha:** If `echo $KRB5CCNAME` is set, make sure the cache file is readable by the user that will run Java. Different shells/sessions sometimes point to different caches.

---

## Step 2 — Confirm the Sybase SPN is resolvable

```bash
kvno sybase/sybase-host.fqdn@YOUR.REALM
klist
```

**✅ Validate:** `kvno` returns a number (e.g. `kvno = 3`) and the second `klist` shows a new service ticket for `sybase/sybase-host.fqdn@YOUR.REALM`.

**⚠️ If it fails with "Server not found in Kerberos database":** the SPN string is wrong. Ask your DBA for the exact registered SPN — that exact string goes into your JDBC connection properties later.

---

## Step 3 — Verify forward and reverse DNS

```bash
host sybase-host.fqdn
host <ip-address-from-above>
```

**✅ Validate:** Forward lookup returns an IP; reverse lookup of that IP returns the same FQDN.

**⚠️ Gotcha:** Kerberos is strict about hostname canonicalization. A reverse-DNS mismatch is the #1 cause of "works in `isql`, fails in Java" — Java is less forgiving than the C client.

---

## Step 4 — Inspect the existing `/etc/krb5.conf`

```bash
ls -l /etc/krb5.conf
cat /etc/krb5.conf
```

**✅ Validate:**
- File exists and is world-readable
- `[libdefaults]` has the correct `default_realm`
- `[realms]` block lists at least one `kdc =` entry for your realm
- `[domain_realm]` maps the Sybase host's domain to the realm

**⚠️ Do not edit this file yourself** on a shared host. If something is wrong (commonly `rdns = true` or missing AES enctypes), raise it with your sysadmin.

For reference, a known-good template looks like:

```ini
[libdefaults]
    default_realm = YOUR.REALM
    dns_lookup_realm = false
    dns_lookup_kdc = false
    ticket_lifetime = 24h
    renew_lifetime = 7d
    forwardable = true
    rdns = false
    udp_preference_limit = 1

[realms]
    YOUR.REALM = {
        kdc = kdc1.your.realm:88
        kdc = kdc2.your.realm:88
        admin_server = kdc1.your.realm:749
        default_domain = your.realm
    }

[domain_realm]
    .your.realm = YOUR.REALM
    your.realm  = YOUR.REALM
    .sybase-host.fqdn = YOUR.REALM
    sybase-host.fqdn  = YOUR.REALM
```

## Step 5 — Create the JAAS login configuration

This file belongs to your application. Create `/path/to/jaas.conf`:

```
SybaseJDBC {
    com.sun.security.auth.module.Krb5LoginModule required
    useTicketCache=true
    doNotPrompt=true
    renewTGT=true
    debug=true;
};
```

**✅ Validate:**
```bash
cat /path/to/jaas.conf
ls -l /path/to/jaas.conf
```
File is readable by the user running the Java app.

**⚠️ Note:** The entry name `SybaseJDBC` is what we pass to jConnect. If jConnect complains about no LoginModule found, try `JaasClient` as the entry name — some versions expect that.

---

## Step 6 — Confirm jConnect version and Kerberos support

```bash
unzip -p /path/to/jconn4.jar META-INF/MANIFEST.MF | head -20
unzip -l /path/to/jconn4.jar | grep -iE "kerb|gss"
```

**✅ Validate:**
- Manifest shows a recent jConnect version (7.07 EBF or later strongly recommended)
- Listing shows classes referring to Kerberos/GSS

---

## Step 7 — Write a minimal Java test program

Create `KrbTest.java`:

```java
import java.sql.*;
import java.util.Properties;

public class KrbTest {
    public static void main(String[] args) throws Exception {
        Class.forName("com.sybase.jdbc4.jdbc.SybDriver");
        String url = "jdbc:sybase:Tds:sybase-host.fqdn:5000/your_db";

        Properties p = new Properties();
        p.setProperty("REQUEST_KERBEROS_SESSION", "true");
        p.setProperty("SERVICE_PRINCIPAL_NAME",
                      "sybase/sybase-host.fqdn@YOUR.REALM");

        try (Connection c = DriverManager.getConnection(url, p);
             Statement s = c.createStatement();
             ResultSet r = s.executeQuery("select suser_name(), @@version")) {
            while (r.next()) {
                System.out.println(r.getString(1) + " | " + r.getString(2));
            }
        }
    }
}
```

Compile:

```bash
javac -cp /path/to/jconn4.jar KrbTest.java
```

**✅ Validate:** `KrbTest.class` is produced with no errors.

## Step 8 — Run with full Kerberos debug

```bash
java \
  -Djava.security.auth.login.config=/path/to/jaas.conf \
  -Djava.security.krb5.conf=/etc/krb5.conf \
  -Djavax.security.auth.useSubjectCredsOnly=false \
  -Dsun.security.krb5.debug=true \
  -Dsun.security.jgss.debug=true \
  -cp .:/path/to/jconn4.jar \
  KrbTest
```

**✅ Validate the output in order:**

1. **JAAS loaded:** `Krb5LoginModule` messages, "Acquire TGT from Cache"
2. **Cache read:** your principal name appears in the debug output
3. **Service ticket requested:** lines mentioning `getting credentials` for `sybase/...`
4. **GSS context established:** "GSSContext established" or similar
5. **Query result printed:** `suser_name()` returns your Kerberos principal — end-to-end success

## Common Failure Modes

| Symptom | Likely cause | Fix |
|---|---|---|
| `No LoginModule found for SybaseJDBC` | JAAS entry name mismatch | Rename entry to `JaasClient` or check jConnect docs for your version |
| `Unable to obtain Principal Name for authentication` | Missing `-Djavax.security.auth.useSubjectCredsOnly=false`, or wrong `KRB5CCNAME` | Add the flag; verify cache path |
| `Server not found in Kerberos database` | SPN string mismatch | Recheck SPN with `kvno`; ensure URL property matches exactly |
| `Defective token detected` / `Checksum failed` | DNS canonicalization or clock skew | Verify forward/reverse DNS; check `chronyc tracking` |
| `KDC has no support for encryption type` | Enctype mismatch with Sybase keytab | Ask DBA which enctypes are in the keytab; add to `krb5.conf` |
| `No valid credentials provided (Mechanism level: ...)` | TGT expired, or Java reading a different cache than your shell | `kinit` again; check `KRB5CCNAME` in Java's environment |
| Works with `isql`, fails in Java | Almost always: `rdns`, enctype, or Java reading a different `krb5.conf` | Force `-Djava.security.krb5.conf=/etc/krb5.conf`; set `rdns = false` |

---

## Quick Reference — Java System Properties

| Property | Purpose |
|---|---|
| `java.security.auth.login.config` | Path to your JAAS config |
| `java.security.krb5.conf` | Path to krb5.conf (override system default) |
| `javax.security.auth.useSubjectCredsOnly=false` | Allow Java to use the ambient ticket cache |
| `sun.security.krb5.debug=true` | Krb5 protocol debug output |
| `sun.security.jgss.debug=true` | GSS-API debug output |

## What to Gather if You Need Help

If something fails after Step 8, capture:

1. Full stderr from the debug-flagged run (Step 8)
2. `klist` output run immediately before the Java command, in the same shell
3. `kvno sybase/sybase-host.fqdn@YOUR.REALM` output
4. Contents of your JAAS config
5. Contents of `/etc/krb5.conf` (redact KDC hostnames if needed but keep structure)
6. `java -version` output
7. jConnect version from Step 6
